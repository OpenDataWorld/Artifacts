# Single Truth, Multiple Native Versions

Status: Normative Foundation Principle

## Principle

> The Fabric preserves one canonical truth while allowing multiple native versions of that truth for every entity, domain, environment, and communication surface.

The canonical truth is not copied blindly into every layer. Each entity receives a version reconstructed for its own identity, context, contract, language, jurisdiction, capability, maturity, and surface. The versions may differ in representation, scope, and delivery method, but they remain linked to one authoritative provenance chain.

## Core Rule

> One truth in the Fabric. Many native representations. One contract lineage. No silent divergence.

## Canonical Model

```text
Canonical Truth
    ↓
Fabric Contract and Provenance
    ↓
Entity Context
    ↓
Native Version Reconstruction
    ↓
Communication Surface
    ↓
Delivered Knowledge
    ↓
Feedback and Reconciliation
```

## Single Truth

Single truth means there is one authoritative, contract-bound record for the current accepted state of a subject.

It preserves:

- canonical identity;
- source;
- owner;
- accountable principal;
- context;
- accepted facts;
- uncertainty;
- version;
- state;
- contract;
- provenance;
- lifecycle;
- correction history.

```text
Canonical Truth
=
Accepted State
+ Contract
+ Provenance
+ Accountability
```

Single truth does not mean one universal presentation or one centralized database.

## Multiple Native Versions

Each entity may receive a native version adapted to:

- role;
- domain;
- language;
- jurisdiction;
- device;
- accessibility needs;
- maturity;
- authority;
- data rights;
- operating mode;
- local infrastructure;
- delivery preference.

Examples include:

- a human-readable explanation;
- a machine-readable contract;
- an agent instruction;
- a dashboard view;
- a local-language version;
- a privacy-minimized projection;
- an edge-optimized representation;
- a hardware-driver configuration;
- a statutory reporting view.

```text
Native Version
=
Canonical Truth
+ Entity Context
+ Surface Contract
+ Minimum Necessary Transformation
```

## Native to Every Entity

Every entity has its own native home in the Fabric.

An entity may be:

- a person;
- an agent;
- an organization;
- a nation;
- a platform;
- a data product;
- a model;
- a tool;
- a server;
- a hardware device;
- a container;
- a Kube;
- a communication surface.

Each entity resolves the truth through its own registered identity and local contract.

## Communication Surface as Contract Holder

The communication surface holds the delivery contract for the interaction it serves.

It must define:

- sender;
- recipient;
- purpose;
- source truth version;
- transformation rules;
- language and format;
- rights and disclosure limits;
- expected acknowledgement;
- delivery evidence;
- expiry;
- correction and withdrawal path.

```text
Communication Surface
=
Delivery Contract Holder
not
Owner of the Underlying Truth
```

The surface may transform presentation, but it may not silently change meaning, authority, provenance, or ownership.

## Knowledge Preservation Across the Stack

The knowledge chain must remain intact across:

```text
Experience Surface
→ Application
→ Agent Runtime
→ Data Model
→ Data Product
→ Fabric Contract
→ Storage Provider
→ Runtime
→ Kernel
→ Hardware
```

Every layer should preserve the minimum metadata required to trace:

- what was known;
- where it came from;
- which version was used;
- which transformation occurred;
- who authorized it;
- what outcome resulted.

## Native Home for Every Component

Every component must be placed in the environment where it is most native and effective.

The platform should match:

- workload to compute;
- data to storage;
- model to accelerator;
- service to runtime;
- policy to enforcement point;
- identity to trust domain;
- experience to user surface;
- driver to hardware;
- state to the appropriate data provider.

```text
Native Placement
=
Best Contextual Fit
+ Registered Environment
+ Known Capability
+ Contracted Responsibility
```

## Metal-Native Kernel

The lowest operational layer must remain grounded in real hardware.

A metal-native kernel provides:

- known processor and architecture;
- registered drivers;
- measurable memory and storage;
- network identity;
- hardware security capabilities;
- resource isolation;
- observable health;
- deterministic interfaces;
- recovery and lifecycle controls.

```text
Metal-Native Kernel
=
Known Hardware Reality
+ Stable Kernel Contract
+ Registered Drivers
+ Observable State
```

Higher layers may remain portable, but production eligibility is always validated against the actual substrate.

## Version Relationships

Every native version must identify its relationship to the canonical truth through fields such as:

```text
derivedFrom
represents
translatedFrom
projectedFrom
redactedFrom
supersedes
validFor
```

A native version may not claim canonical status unless the Fabric gate explicitly accepts it as the new authoritative state.

## No Silent Divergence

When a native version differs materially from the canonical truth, the platform must:

```text
Detect
→ Mark Difference
→ Preserve Both Versions
→ Identify Cause
→ Review Contract and Context
→ Reconcile or Reject
```

Conflicts become visible evidence. They are never silently merged.

## Shared Context with Separation of Concern

The Fabric shares enough context to maintain coherence while preserving local ownership and concern boundaries.

```text
Canonical Truth
+ Bounded Context Projection
+ Native Reconstruction
=
Coherent Distributed Knowledge
```

Shared context never grants universal authority or unrestricted disclosure.

## Headless and Universal Distribution

The same canonical truth may be delivered through many surfaces:

- web;
- mobile;
- API;
- CLI;
- voice;
- messaging;
- dashboard;
- document;
- event stream;
- agent inbox;
- embedded device.

Each surface holds its own delivery contract while preserving the same truth lineage.

```text
One Canonical Truth
+ Many Surface Contracts
=
Universal Distribution Without Meaning Loss
```

## Platform Responsibilities

The platform must:

- maintain canonical truth identities;
- preserve immutable provenance;
- register all entities and environments;
- reconstruct native versions;
- validate transformations;
- bind communication surfaces to delivery contracts;
- detect semantic drift;
- preserve ownership and accountability;
- recommend native placement for every component;
- validate the metal-native substrate;
- support correction, supersession, migration, and exit.

## Platform Invariants

1. Every subject has one accepted canonical truth state at a time.
2. Every entity may receive a context-native version of that truth.
3. Native versions preserve a complete relationship to the canonical source.
4. Communication surfaces hold delivery contracts, not ownership of truth.
5. Presentation may change; meaning and provenance may not change silently.
6. Every component has a registered native home in the stack.
7. Production execution is grounded in a metal-native kernel and registered hardware.
8. Conflicting versions remain visible and trigger reconciliation.
9. No operation or transformation occurs in the dark.
10. The Fabric preserves knowledge continuity across every layer and provider.

## Final Statement

> Preserve one canonical truth in the Fabric and reconstruct multiple native versions for every entity, domain, environment, and communication surface. Let each surface hold the contract for how knowledge is communicated, while the Fabric preserves meaning, ownership, accountability, and provenance across the full stack. Give every component its native home, grounded ultimately in a registered metal-native kernel, so distribution can remain universal without fragmenting truth.
