# AnyDB Kube Contract v0.1

Status: Draft

## Purpose

The AnyDB Kube Contract defines the smallest complete, governable, and reconcilable unit of data in the AnyDB platform.

A Kube is the canonical boundary that binds identity, state, time, location, policy, evidence, contract, events, projections, lifecycle, and reconciliation into one durable unit.

## Core Rule

> A Kube is complete only when it can be identified, governed, transformed, moved, executed, observed, and reconciled without losing meaning.

## Dimensions

### Time

A Kube should preserve the relevant time dimensions:

- valid time
- transaction time
- occurrence time
- observation time
- execution time
- reconciliation time

### Location

A Kube should preserve the relevant location dimensions:

- physical location
- logical location
- authority location
- infrastructure location
- execution location
- observation location
- jurisdiction

Unknown dimensions must be explicit.

## `spec.state`

Contains the current authoritative domain state.

Rules:

- validated against the declared schema
- versioned
- committed through a declared transaction boundary
- historically reconstructable
- never silently overwritten by a projection

## `spec.relationships`

Typed links to other Kubes or canonical entities.

Example:

```yaml
relationships:
  - type: belongsTo
    target: urn:anydb:kube:organization:opendataworld
    validFrom: 2026-06-11T00:00:00Z
    validTo: null
```

## `spec.policies`

References policies governing access, mutation, retention, placement, execution, and transformation.

## `spec.contract`

References the platform-held contract that defines:

- allowed state transitions
- required evidence
- accepted providers
- completion rules
- consistency expectations
- portability and exit

## `spec.evidence`

References source material, attestations, approvals, and proofs supporting the Kube state or lifecycle.

## `spec.artifacts`

References durable artifacts attached to or represented by the Kube.

Examples:

- OCI images
- models
- documents
- datasets
- policies
- schemas
- signatures

## `status.provenance`

Records source, transformation, actor, runtime, schema, and version lineage.

## `status.events`

References immutable events describing the Kube's lifecycle and changes.

## `status.projections`

Declares derived representations such as:

- record view
- document view
- graph view
- search view
- vector view
- feature view
- edge replica

Every projection must identify the source Kube version.

## `status.reconciliation`

Recommended states:

```text
unknown
pending
converged
diverged
compensating
blocked
failed
```

Terminal outcomes may include:

```text
converged
compensated
rejected
expired
cancelled
failed-with-evidence
escalated-to-human
```

## `status.trust`

Carries evidence-backed trust state.

Recommended values:

```text
unknown
provisional
verified
trusted
degraded
disputed
suspended
revoked
```

## Atomicity

A valid authoritative Kube transition should atomically preserve:

1. the new Kube state
2. the incremented version
3. the transition event
4. actor identity
5. policy decision
6. durable projection intent
7. reconciliation requirement

## Reactive Behavior

After an ACID commit, a Kube may reactively trigger:

- projections
- notifications
- agent context updates
- edge commands
- model inference
- reconciliation

Events may trigger commands, but must not bypass policy or mutation validation.

## Lossless Transformation

Any Kube transformation must be:

```text
lossless-reversible
lossless-source-retained
lossless-source-referenced
lossy-declared
```

A lossy transformation cannot silently replace the authoritative Kube.

## Edge Operation

A Kube may be admitted to an edge runtime only when the edge can verify:

- Kube identity
- contract version
- policy version
- artifact digests
- workload identity
- data placement rules

The edge returns:

- observed state
- execution evidence
- local events
- local version
- reconciliation result

## Composition

Kubes compose without losing individual identity.

```text
Kube
  ↓
KubeSet
  ↓
KubeGraph
  ↓
Fabric
```

### KubeSet

A governed collection of Kubes sharing a purpose or contract.

### KubeGraph

Typed relationships between Kubes.

### Fabric

A continuously reconciled network of Kubes, contracts, identities, providers, and outcomes.

## Provider Obligations

A provider implementing Kube storage or execution must declare:

- canonical ID mapping
- transaction semantics
- temporal behavior
- relationship behavior
- event handling
- projection handling
- deletion and retention
- export and restore
- provider-specific extensions

## Validation Invariants

A Kube is invalid when any of the following are true:

1. canonical ID is missing or mutable
2. schema cannot be resolved
3. version regresses
4. lifecycle state is unknown
5. relevant dimensions are omitted without explicit unknown status
6. policy references are invalid
7. contract is missing for high-consequence work
8. projection source version is absent
9. provenance required by the source is absent
10. reconciliation claims completion without a terminal outcome
11. provider extensions overwrite core semantics
12. a lossy view presents itself as authoritative without promotion

## Surface Obligations

### Sources

Create or reference source Kubes with source identity and observation dimensions.

### Ingest

Map incoming data into Kube envelopes without losing source meaning.

### Store

Persist authoritative Kube state, history, events, and versions.

### Search

Expose searchable Kube projections with source-version traceability.

### Intelligence

Consume Kube context and return evidence-linked observations or proposals.

### Platform

Register Kube schemas, policies, contracts, providers, and lifecycle state.

### Edge

Execute admitted Kube contracts and return observed evidence.

### Assurance

Test Kube conformance, portability, transformation, and reconciliation.

### Artifacts

Publish Kube schemas, examples, migrations, and compatibility guidance.

## Final Invariant

> Every Kube must remain one bounded unit of identity, truth, context, contract, and lifecycle across all representations, providers, channels, and execution locations.
