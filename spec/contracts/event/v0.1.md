# AnyDB Event Contract v0.1

Status: Draft

## Purpose

The AnyDB Event Contract defines the minimum interoperable envelope for recording facts that occur within or around the AnyDB platform.

Events provide the causal history behind entity state, projections, workflows, policy decisions, external effects, and reconciliation.

## Core Rule

> An AnyDB event records an immutable fact that has occurred. It does not represent an unverified intention.

Commands express intent. Events record accepted outcomes or observed facts.

## Contract Goals

The contract provides:

- globally unique event identity
- event type and schema
- canonical subject identity
- actor identity
- occurrence and recording time
- causation and correlation
- policy and decision references
- evidence and provenance
- payload integrity
- delivery and processing metadata
- reconciliation context

## Canonical Envelope

```yaml
apiVersion: anydb.opendata.world/v1alpha1
kind: Event
metadata:
  id: urn:anydb:event:01JZEXAMPLE
  type: CustomerPlanChanged
  schema:
    id: urn:anydb:schema:event:customer-plan-changed
    version: 1.0.0
  subject: urn:anydb:customer:42
  actor: urn:anydb:agent:billing-operator
  occurredAt: 2026-06-11T09:29:58Z
  recordedAt: 2026-06-11T09:30:00Z
  sequence: 19
  correlationId: urn:anydb:workflow:order-8821
  causationId: urn:anydb:command:01JZCOMMAND
  tenant: urn:anydb:organization:acme
spec:
  data: {}
  evidence: []
  policyDecision: urn:anydb:decision:01JZPOLICY
  relationships: []
status:
  provenance: []
  delivery: {}
  reconciliation: {}
  trust: {}
```

## Required Fields

### `apiVersion`

Required value for this draft:

```text
anydb.opendata.world/v1alpha1
```

### `kind`

Required value:

```text
Event
```

### `metadata.id`

Globally unique, immutable event identifier.

Recommended form:

```text
urn:anydb:event:<sortable-identifier>
```

Implementations should prefer identifiers that preserve approximate creation order without treating the identifier itself as authoritative time.

### `metadata.type`

The semantic event type.

Examples:

```text
EntityCreated
CustomerPlanChanged
ArtifactPublished
PolicyEvaluated
AgentSuspended
ProjectionRebuilt
ReconciliationFailed
```

Event types should use past-tense facts.

### `metadata.schema`

Declares the schema governing the event payload.

Required members:

- `id`
- `version`

Consumers must validate or explicitly quarantine events with unsupported schemas.

### `metadata.subject`

The canonical identity of the primary entity affected or described by the event.

An event may reference additional entities through relationships, but one subject should remain primary.

### `metadata.occurredAt`

The time when the event happened in the business or observed world.

### `metadata.recordedAt`

The time when AnyDB durably recorded the event.

Occurrence time and recording time must remain distinct.

### `metadata.sequence`

The monotonic sequence number within the subject stream or declared aggregate boundary.

This supports ordering and optimistic concurrency.

Global ordering across unrelated subjects is not required unless explicitly provided by a provider.

## Actor

### `metadata.actor`

The canonical identity responsible for or associated with the event.

Possible actor types include:

- person
- service
- agent
- controller
- external system
- unknown observer

System-generated events must still identify the producing component.

## Causality

### `metadata.causationId`

References the direct command, event, or observation that caused this event.

### `metadata.correlationId`

Groups events belonging to one workflow, transaction, request, incident, or business process.

Example:

```text
Command A
  -> Event B
      -> Event C
      -> Projection D
```

Each event should preserve causation, while all may share the same correlation ID.

## Tenant and Scope

### `metadata.tenant`

Canonical tenant or organizational boundary.

Cross-tenant events require explicit federation policy and provenance.

## `spec.data`

Contains the event-specific immutable payload.

Rules:

- validated against the declared event schema
- should capture facts needed for reconstruction and audit
- should avoid unnecessary duplication of sensitive entity data
- should prefer canonical identity references over embedded copies

Example:

```yaml
data:
  previousPlan: standard
  newPlan: enterprise
  effectiveFrom: 2026-07-01T00:00:00Z
```

## `spec.evidence`

References evidence supporting the event.

Example:

```yaml
evidence:
  - id: urn:anydb:artifact:approved-contract-778
    relation: authorizes
    digest: sha256:...
```

## `spec.policyDecision`

References the policy decision that permitted, denied, constrained, or classified the action.

Events representing denied commands may reference the denial decision.

## `spec.relationships`

Additional canonical identities related to the event.

Example:

```yaml
relationships:
  - type: initiatedBy
    target: urn:anydb:user:17
  - type: affectedResource
    target: urn:anydb:subscription:991
```

## `status.provenance`

Records where the event originated and how it entered AnyDB.

Example:

```yaml
provenance:
  - source: urn:anydb:source:billing-system
    sourceEventId: evt_8127
    observedAt: 2026-06-11T09:29:59Z
    ingestedAt: 2026-06-11T09:30:00Z
    transform: urn:anydb:transform:billing-event:v1
```

## `status.delivery`

Tracks delivery metadata without changing the immutable event fact.

Example:

```yaml
delivery:
  attempts: 2
  lastAttemptAt: 2026-06-11T09:30:03Z
  consumers:
    - name: semantic-search
      checkpoint: 19
      state: acknowledged
    - name: feature-store
      checkpoint: 18
      state: pending
```

Delivery state is mutable operational metadata. The event payload remains immutable.

## `status.reconciliation`

Connects the event to verification or repair workflows.

Example:

```yaml
reconciliation:
  required: true
  state: pending
  controller: urn:anydb:controller:billing-reconciler
```

Recommended states:

```text
not-required
pending
verified
diverged
compensating
resolved
failed
```

## `status.trust`

Carries verification metadata about the event source, signature, and evidence.

Example:

```yaml
trust:
  verification: verified
  signature:
    algorithm: ed25519
    keyId: urn:anydb:key:billing-system:1
    value: base64:...
```

Trust metadata cannot replace policy evaluation.

## Event Immutability

After an event is accepted into the authoritative event store, these fields must not change:

- event ID
- event type
- schema
- subject
- actor
- occurred time
- recorded time
- sequence
- causation
- correlation
- payload
- evidence references present at commit

Operational status fields may evolve independently.

Corrections must be represented by new events.

Example:

```text
CustomerAddressRecorded
CustomerAddressCorrected
```

The original event remains preserved.

## Ordering

AnyDB guarantees ordering only within an explicitly declared stream boundary.

Recommended boundary:

```text
subject + stream name
```

Example:

```text
urn:anydb:customer:42 / lifecycle
```

Consumers must not infer global causal order from timestamps alone.

## Duplicate Delivery

Events may be delivered more than once.

Consumers must process events idempotently using:

- event ID
- subject sequence
- consumer checkpoint
- deduplication store

## At-Least-Once Delivery

The default AnyDB event-delivery assumption is at-least-once unless a provider explicitly guarantees otherwise.

Exactly-once business semantics should be achieved through idempotent processing and transactionally maintained checkpoints.

## Transactional Commit

When an entity mutation and event are part of one authoritative change, they must be committed atomically within the provider boundary.

If event publication to external systems occurs asynchronously, the outbox pattern or equivalent durable mechanism should be used.

## Command Relationship

A command is not an event.

Command example:

```text
ChangeCustomerPlan
```

Accepted event:

```text
CustomerPlanChanged
```

Rejected event:

```text
CustomerPlanChangeRejected
```

The platform should never record intent as completed fact before execution succeeds.

## Observation Events

External observations may be recorded even when AnyDB does not control the source.

Examples:

```text
ExternalPaymentObserved
ProviderStateChanged
FileDiscovered
TelemetryReceived
```

Observation events must preserve source, observation time, ingestion time, and confidence.

## Policy Events

Policy evaluation may produce events such as:

```text
PolicyEvaluationCompleted
AccessDenied
ApprovalRequired
ConstraintViolated
```

These events should reference the policy version, decision identity, actor, resource, action, and evidence.

## Reconciliation Events

Recommended reconciliation event types:

```text
ReconciliationStarted
ObservedStateMatched
DivergenceDetected
RetryScheduled
CompensationStarted
CompensationCompleted
ReconciliationResolved
ReconciliationFailed
```

## Projection Events

Recommended projection lifecycle event types:

```text
ProjectionRequested
ProjectionStarted
ProjectionCompleted
ProjectionFailed
ProjectionInvalidated
ProjectionRebuilt
```

Each should reference the source entity version and projection version.

## Deletion Events

Deletion must be explicit.

Examples:

```text
EntityDeletionRequested
EntityTombstoned
EntityRedacted
EntityPhysicallyErased
ProjectionDeletionConfirmed
```

The event stream should preserve legal and operational evidence without reproducing data that must be erased.

## Event Schema Evolution

Event schemas are immutable once published.

Compatible change:

- adding optional fields
- expanding allowed enum values when consumers can safely ignore unknown values

Breaking change:

- removing fields
- changing field meaning
- changing requiredness
- changing identity semantics

Breaking changes require a new schema version or event type.

## Minimal Valid Event

```yaml
apiVersion: anydb.opendata.world/v1alpha1
kind: Event
metadata:
  id: urn:anydb:event:01JZEXAMPLE
  type: ArtifactPublished
  schema:
    id: urn:anydb:schema:event:artifact-published
    version: 1.0.0
  subject: urn:anydb:artifact:acid-vs-base
  actor: urn:anydb:user:chinmay-panda
  occurredAt: 2026-06-11T10:00:00Z
  recordedAt: 2026-06-11T10:00:01Z
  sequence: 1
  correlationId: urn:anydb:workflow:publish-artifact-1
  causationId: urn:anydb:command:publish-artifact-1
  tenant: urn:anydb:organization:opendataworld
spec:
  data:
    title: ACID vs BASE
  evidence: []
  policyDecision: null
  relationships: []
status:
  provenance: []
  delivery: {}
  reconciliation:
    required: false
    state: not-required
  trust: {}
```

## Validation Invariants

An event is invalid when any of the following are true:

1. event ID is missing or reused
2. type is not a past-tense fact
3. schema cannot be resolved
4. subject lacks canonical identity
5. occurred time is missing
6. recorded time is missing
7. sequence regresses within the stream
8. causation references itself
9. payload fails schema validation
10. a previously committed immutable field is changed
11. provenance required by the source contract is absent
12. tenant boundary is violated without federation policy

## Surface Obligations

### Sources

Must preserve external event IDs, source identity, observation time, and source ordering when available.

### Ingest

Must validate, normalize, deduplicate, and preserve lineage.

### Store

Must preserve immutability, sequence, atomic commit, and durable retrieval.

### Search

Must consume events idempotently and maintain source-version checkpoints.

### Features

Must preserve point-in-time correctness and event lineage.

### Platform

Must expose event registration, routing, status, and replay controls.

### Assurance

Must test duplicate delivery, ordering boundaries, schema compatibility, replay, and failure recovery.

### Artifacts

Must publish event schemas, examples, migration guidance, and compatibility history.

## Final Invariant

> Every accepted AnyDB event must remain an immutable, causally linked, policy-aware, and provenance-preserving fact that can be replayed, audited, projected, and reconciled.
