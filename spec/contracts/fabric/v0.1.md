# AnyDB Fabric Contract v0.1

Status: Draft

## Purpose

The AnyDB Fabric Contract defines how Kubes compose into a continuously reconciled graph of identities, relationships, contracts, events, policies, providers, artifacts, decisions, and outcomes.

A Kube is the smallest complete unit of governed truth.

The Fabric is the living network that connects those Kubes and keeps their relationships, obligations, projections, and observed outcomes coherent over time and location.

## Core Rule

> The Fabric is a continuously reconciled graph of Kubes governed by platform-held contracts.

## Canonical Model

```text
Kube
  ↓
KubeSet
  ↓
KubeGraph
  ↓
Fabric
```

### Kube

Smallest complete governed unit.

### KubeSet

A bounded collection of Kubes sharing a contract, purpose, tenant, domain, or lifecycle.

### KubeGraph

Typed, temporal, policy-aware relationships between Kubes.

### Fabric

The federated, continuously reconciled network of KubeGraphs, providers, channels, and execution outcomes.

## Fabric Identity

Every Fabric has one canonical identity.

A Fabric identity defines the governance boundary for:

- canonical namespaces
- contract ownership
- policy authority
- provider registration
- federation
- reconciliation
- trust evaluation

A deployment, tenant, region, or edge may host a Fabric partition or replica without becoming an unrelated Fabric.

## KubeSet Contract

A KubeSet defines a bounded collection.

```yaml
apiVersion: anydb.opendata.world/v1alpha1
kind: KubeSet
metadata:
  id: urn:anydb:kubeset:published-artifacts
  version: 1
spec:
  selector:
    types:
      - ArtifactKube
    labels:
      publicationStatus: published
  contract: urn:anydb:contract:artifact-publication:v1
  policy: urn:anydb:policy:public-artifact-read:v1
  consistency: eventual
  placement:
    mode: global-replicated
```

A KubeSet must declare:

- membership rule
- contract
- policy
- consistency expectations
- placement
- lifecycle
- reconciliation behavior

## KubeGraph Contract

A KubeGraph defines typed relationships between Kubes.

```yaml
apiVersion: anydb.opendata.world/v1alpha1
kind: KubeGraph
metadata:
  id: urn:anydb:kubegraph:anydb-platform
  version: 1
spec:
  nodes:
    source: anydb-kubes
  edgeSchemas:
    - type: belongsTo
      temporal: true
    - type: producedBy
      temporal: true
    - type: governedBy
      temporal: true
    - type: deployedAt
      temporal: true
  policy: urn:anydb:policy:fabric-graph-access:v1
```

Every graph relationship must preserve:

- source Kube ID
- target Kube ID
- relationship type
- valid time
- transaction time
- provenance
- policy context
- evidence where required

## Fabric Planes

### Contract Plane

Defines obligations, completion rules, provider capabilities, and exit paths.

### Identity Plane

Resolves every actor, owner, provider, edge, agent, and artifact.

### Data Plane

Stores authoritative Kube state and event history.

### Relationship Plane

Maintains typed, temporal graph relationships.

### Projection Plane

Produces search, vector, graph, feature, and analytical views.

### Execution Plane

Carries governed commands to providers, agents, applications, and edges.

### Reconciliation Plane

Compares intended, authoritative, projected, observed, and delivered state.

### Trust Plane

Evaluates evidence-backed conformance over time.

## Fabric State Model

The Fabric distinguishes:

```text
Intended State
Authoritative State
Projected State
Observed State
Delivered State
Reconciled State
```

### Intended State

What a contract, command, user, controller, or agent requests.

### Authoritative State

What AnyDB has accepted as canonical truth.

### Projected State

What derived surfaces currently expose.

### Observed State

What external providers, edges, or sensors report.

### Delivered State

What a partner, channel, provider, or runtime claims to have completed.

### Reconciled State

The verified state after comparison, repair, compensation, or escalation.

## Fabric Reconciliation Loop

```text
Contract
  ↓
Desired State
  ↓
Command
  ↓
Provider / Agent / Edge Execution
  ↓
Observed State
  ↓
Evidence
  ↓
Reconciliation
  ├── converged
  ├── retry
  ├── compensate
  ├── quarantine
  ├── escalate
  └── failed-with-evidence
```

The loop is continuous because providers, projections, relationships, and external state may drift after initial convergence.

## Relationship Reconciliation

The Fabric must reconcile not only object state but also graph relationships.

Examples:

- an agent no longer belongs to an organization
- a provider capability has expired
- an artifact was rebuilt from a new source version
- a credential was revoked
- an edge moved to another jurisdiction
- a partner lost certification

Relationship drift can change policy outcomes and must be treated as operationally significant.

## Contract Propagation

A platform-held contract may be distributed to providers and edges as a signed artifact.

The distributed contract must preserve:

- contract ID
- version
- owner
- policy references
- obligations
- completion rules
- evidence requirements
- validity interval
- signature or integrity digest

Local runtimes may not silently reinterpret the contract.

## Fabric Queries

The Fabric should support governed queries such as:

- Which Kubes are governed by this policy?
- Which artifacts were produced by this model version?
- Which partner delivered this edge deployment?
- Which edges are running a superseded contract?
- Which projections lag the canonical Kube version?
- Which identities were accountable when this event occurred?
- Which obligations remain unresolved?

## Federation

A Fabric may federate with another Fabric without collapsing governance domains.

Federation must preserve:

- home authority
- canonical namespaces
- contract ownership
- policy boundaries
- provenance
- trust assertions
- revocation
- data-placement constraints
- reconciliation responsibilities

```text
Fabric A
   |
Federation Contract
   |
Fabric B
```

Cross-Fabric access and relationships require an explicit federation contract.

## Edge Fabric

An edge may run a local Fabric partition containing:

- admitted Kubes
- local policies
- signed contracts
- local event log
- local projections
- intelligence providers
- reconciliation controller

During disconnection, the edge continues within its delegated contract.

After reconnection, the local and wider Fabric exchange events, evidence, and reconciliation results.

## Partner and Channel Fabric

Partners and channels participate as identified KubeGraphs, not as ungoverned external actors.

The Fabric records:

- partner identity
- certification
- contract scope
- delivered artifacts
- execution locations
- evidence
- breaches
- reconciliation history

## Supply Chain Fabric

Every source, build, model, artifact, registry, deployment, partner, edge, and outcome can be represented as connected Kubes.

```text
Source Kube
   ↓ produced
Build Kube
   ↓ produced
Artifact Kube
   ↓ delivered-by

   ↓ deployed-at

   ↓ produced
```

This makes the supply chain queryable and governable as one graph.

Trust may be evaluated at the level of:

- Kube

## Fabric Health

Recommended health states:

```text
healthy
degraded
diverged
partitioned
compensating
blocked
failed
```

Fabric health must include evidence and scope. One stale projection should not necessarily mark the entire Fabric failed.

## Consistency

The Fabric does not impose one global consistency model.

Each KubeSet, relationship type, provider, and operation declares consistency requirements.

Strong coordination is recommended for:

- canonical identity
- contract activation
- policy assignment
- ownership
- financial commitments
- irreversible actions

Eventual consistency is acceptable for:

- search
- vectors
- analytics
- telemetry
- low-consequence replication

## Lossless Transformation

Every representation and movement across the Fabric must preserve Kube identity, source version, lineage, and declared loss behavior.

A transformation may change form without severing the path to source truth.

## Portability

The Fabric contract must remain implementable across:

- databases
- graph engines
- event systems
- clouds
- Kubernetes distributions
- edges
- model runtimes
- identity systems
- policy engines

No provider owns Fabric semantics.

## Validation Invariants

A Fabric is invalid or degraded when any of the following are true:

1. a Kube cannot resolve to canonical identity
2. a relationship has no typed semantics
3. a high-consequence action has no contract
4. a provider claims an undeclared capability
5. a projection loses source-version traceability
6. an edge executes an expired contract
7. a partner reports completion without required evidence
8. reconciliation state is hidden
9. federation discards authority or provenance
10. a lossy representation presents itself as canonical truth

## Surface Obligations

### Foundation

Defines Fabric semantics and invariants.

### Platform

Registers Kubes, contracts, providers, KubeSets, and topology.

### Sources and Ingest

Create traceable source and transformation relationships.

### Store

Persists authoritative Kube state, event streams, and graph edges.

### Search and Intelligence

Produce governed projections and observations linked to Kube versions.

### Edge

Executes delegated contracts and returns evidence.

### Partners and Channels

Deliver obligations under identified contracts.

### Assurance

Tests graph integrity, contract conformance, portability, and reconciliation.

### Artifacts

Publishes schemas, contracts, examples, compatibility, and migrations.

## Platform Invariants

1. The Fabric is composed of canonical Kubes.
2. Every relationship is typed, temporal, and provenance-aware.
3. Every high-consequence action resolves a platform-held contract.
4. Every provider and partner has canonical identity.
5. Every projection points to source Kube versions.
6. Every edge operation produces evidence.
7. Every divergence remains visible until resolved.
8. Federation preserves authority and policy boundaries.
9. Trust is evidence-backed and revisable.
10. The Fabric continuously reconciles truth with reality.
