# AnyDB Entity Contract v0.1

Status: Draft

## Purpose

The AnyDB Entity Contract defines the minimum interoperable envelope for an authoritative entity across all AnyDB surfaces, providers, projections, runtimes, and deployments.

An entity may be represented as a record, document, graph node, event subject, vector source, artifact, time-series subject, or external resource. Those representations must preserve one canonical identity and governance context.

## Core Rule

> One real or logical entity must have one canonical AnyDB identity, even when represented through many models and providers.

## Contract Goals

The contract provides:

- canonical identity
- declared type and schema
- versioning
- ownership
- temporal validity
- provenance
- policy references
- relationships
- trust metadata
- projection status
- reconciliation status

## Canonical Envelope

```yaml
apiVersion: anydb.opendata.world/v1alpha1
kind: Entity
metadata:
  id: urn:anydb:customer:42
  type: Customer
  schema:
    id: urn:anydb:schema:customer
    version: 1.0.0
  version: 18
  owner: urn:anydb:organization:acme
  createdAt: 2026-06-01T10:00:00Z
  updatedAt: 2026-06-11T09:30:00Z
  validFrom: 2026-06-01T00:00:00Z
  validTo: null
  labels:
    domain: customer
    classification: confidential
spec:
  data: {}
  relationships: []
  policies: []
  evidence: []
status:
  provenance: []
  trust: {}
  projections: []
  reconciliation: {}
```

## Required Fields

### `apiVersion`

Identifies the AnyDB contract version used by the serialized entity.

Required value for this draft:

```text
anydb.opendata.world/v1alpha1
```

### `kind`

Identifies the contract object type.

Required value:

```text
Entity
```

### `metadata.id`

The canonical, stable identifier for the entity.

Requirements:

- globally unique within the AnyDB federation
- immutable after canonical assignment
- independent of a provider's internal primary key
- safe to reference from events, policies, relationships, and projections

Recommended form:

```text
urn:anydb:<type>:<identifier>
```

Provider-specific identifiers may be retained as aliases, but must not replace the canonical ID.

### `metadata.type`

The declared semantic type of the entity.

Examples:

```text
Customer
Artifact
Agent
Policy
Organization
Dataset
Decision
```

The type must resolve to a registered schema or type definition.

### `metadata.schema`

Declares the schema governing the entity payload.

Required members:

- `id`
- `version`

The schema version used to validate a historical entity version must remain discoverable.

### `metadata.version`

The authoritative entity version.

Requirements:

- monotonically increasing within the entity history
- incremented for every accepted authoritative change
- usable for optimistic concurrency

A projection version must not be substituted for the authoritative entity version.

### `metadata.owner`

The canonical identity accountable for the entity.

Ownership does not automatically grant unrestricted access. Policy remains authoritative for access decisions.

### `metadata.createdAt`

The transaction time when the canonical entity was first recorded by AnyDB.

### `metadata.updatedAt`

The transaction time of the latest accepted authoritative version.

## Temporal Fields

### `metadata.validFrom`

The time from which the entity version is valid in the business or domain world.

### `metadata.validTo`

The time until which the entity version is valid.

A `null` value indicates an open-ended validity interval.

Temporal intervals should use a half-open convention:

```text
[validFrom, validTo)
```

This prevents overlap at exact transition boundaries.

## `spec.data`

Contains the domain-specific entity payload.

Rules:

- validated against `metadata.schema`
- must not redefine canonical envelope fields with conflicting meaning
- should preserve lossless source values where required
- sensitive fields must carry or inherit classification and policy controls

## `spec.relationships`

Contains typed references to other canonical entities.

Example:

```yaml
relationships:
  - type: belongsTo
    target: urn:anydb:organization:acme
    validFrom: 2026-01-01T00:00:00Z
    validTo: null
    attributes:
      role: enterprise-customer
```

Requirements:

- `target` must be a canonical ID or an unresolved identity reference
- relationships should be typed
- temporal validity should be explicit when relevant
- relationship provenance must be discoverable

## `spec.policies`

References policies applicable to the entity.

Example:

```yaml
policies:
  - urn:anydb:policy:customer-access:v3
  - urn:anydb:policy:retention:seven-years
```

Policy references do not embed a cached allow or deny decision. Runtime access must evaluate the relevant policy version and context.

## `spec.evidence`

References source material or attestations supporting the entity or specific claims.

Example:

```yaml
evidence:
  - id: urn:anydb:artifact:contract-778
    relation: supports
    digest: sha256:...
```

Evidence should remain versioned and verifiable.

## `status.provenance`

Records source and transformation lineage.

Example:

```yaml
provenance:
  - source: urn:anydb:source:crm
    sourceId: customer_42
    observedAt: 2026-06-11T09:29:55Z
    ingestedAt: 2026-06-11T09:30:00Z
    transform: urn:anydb:transform:crm-customer:v2
```

Requirements:

- retain the original source reference
- retain transformation identity and version
- distinguish observation time from ingestion time
- preserve checksums where source integrity matters

## `status.trust`

Carries verification and confidence metadata.

Example:

```yaml
trust:
  verification: verified
  score: 0.94
  assertions:
    - issuer: urn:anydb:organization:acme
      type: source-controlled
```

Trust metadata must not silently override policy.

## `status.projections`

Records known derived representations.

Example:

```yaml
projections:
  - name: semantic-search
    provider: urn:anydb:provider:opensearch
    sourceVersion: 18
    projectionVersion: 44
    projectedAt: 2026-06-11T09:30:04Z
    status: ready
```

A projection must identify the authoritative source version from which it was derived.

## `status.reconciliation`

Describes comparison between authoritative, projected, and externally observed state.

Example:

```yaml
reconciliation:
  state: converged
  checkedAt: 2026-06-11T09:31:00Z
  observedVersion: 18
  conditions: []
```

Recommended states:

```text
unknown
pending
converged
diverged
compensating
blocked
failed
```

## Identity Aliases

Provider and source identifiers may be retained as aliases.

Example:

```yaml
metadata:
  aliases:
    - scheme: crm
      value: customer_42
    - scheme: postgres
      value: public.customers/42
```

Aliases are mutable mappings. The canonical AnyDB ID remains stable.

## Mutation Rules

An authoritative mutation must:

1. authenticate the actor
2. resolve canonical identity
3. validate the requested schema
4. evaluate applicable policy
5. check the expected entity version
6. protect domain invariants
7. commit the new version atomically within the provider boundary
8. record an authoritative event or durable mutation record
9. request projection updates
10. schedule verification or reconciliation when external effects exist

## Optimistic Concurrency

A mutation should carry an expected version.

Example:

```yaml
expectedVersion: 18
```

If the current version is not 18, the mutation must fail with a conflict or enter an explicitly governed merge process.

Silent last-write-wins behavior is not permitted for high-consequence entities unless declared by policy.

## Deletion and Tombstones

Deletion should be explicit.

Recommended representation:

```yaml
metadata:
  lifecycle: deleted
status:
  tombstone:
    deletedAt: 2026-06-11T10:00:00Z
    deletedBy: urn:anydb:user:17
    reason: retention-expired
```

Physical erasure may occur later according to privacy, retention, and legal requirements.

Historical references should resolve to a tombstone or redacted record rather than silently breaking.

## Privacy and Redaction

The entity contract supports redaction and erasure without requiring every historical projection to expose deleted personal data.

Implementations should support:

- field-level redaction
- pseudonymization
- retention expiry
- cryptographic destruction where appropriate
- projection invalidation
- audit evidence that does not reproduce erased content

## Projection Rules

A projection of an entity must preserve:

- canonical entity ID
- authoritative source version
- schema or projection schema version
- projection time
- provenance reference
- policy context or policy reference
- freshness status

A projection must not present itself as authoritative unless it has been explicitly promoted through a governed process.

## Provider Rules

A provider implementation must declare:

- how canonical IDs map to local keys
- supported transaction semantics
- versioning behavior
- temporal capabilities
- relationship capabilities
- deletion behavior
- backup and restore behavior
- consistency limits

Provider-specific extensions must be namespaced.

Example:

```yaml
extensions:
  surrealdb.opendata.world:
    table: customer
```

## Validation Invariants

An entity is invalid when any of the following are true:

1. canonical ID is missing or malformed
2. schema cannot be resolved
3. version is missing or regresses
4. validity interval is inverted
5. a relationship target is ambiguous without unresolved status
6. provenance is required but absent
7. projection claims a source version that does not exist
8. policy references are syntactically invalid
9. provider extensions overwrite core fields
10. the entity payload fails schema validation

  type: Artifact

## Surface Obligations

### Sources

Must preserve source identifiers and observation metadata.

### Ingest

Must validate and map external data without losing provenance.

### Store

Must enforce canonical identity, versioning, and authoritative transaction behavior.

### Search

Must retain canonical ID, source version, freshness, and policy context.

### Features

Must retain entity ID, point-in-time source references, lineage, and feature version.

### Platform

Must register schemas, policies, providers, and reconciliation status.

### Assurance

Must test conformance against this contract.

### Artifacts

Must publish the contract, examples, migration guidance, and version history.

## Compatibility

Additive optional fields may be introduced within the same compatible version.

Breaking changes require:

- a new contract version
- migration guidance
- provider compatibility declarations
- projection rebuild guidance
- release notes

## Final Invariant

> Every AnyDB representation must remain traceable to one canonical entity identity, one declared authoritative version, and one explainable governance history.
